# Lite Atoms Dev 运行时与数据设计

> 版本：3.0  
> 本文是 [PRD](./1-PRD.md) 的可执行运行时契约，配合 [架构总览](./2-设计文档.md) 阅读。它规定状态机、存储事实来源、事务边界、数据流、SSE 和失败语义；不重复定义产品范围。

## 1. 设计结论

初版常见的 `Project + Message + Commit snapshot + Job` 模型不能直接采用：JSONB 快照不是 Git Commit；`agent_jobs` 与 Run 重复；对话、Tool 轨迹与诊断日志混放不安全；Git、Storage 与数据库也不能假装处于同一 ACID 事务。

最终模型是：

```text
Git                = 源码和真实 Commit 的唯一事实来源
Supabase Postgres  = 产品状态、队列、版本索引、事件的唯一事实来源
Supabase Storage   = 不可变 Artifact、长日志、导出文件
agent_runs         = 用户动作 + 队列任务的唯一模型
SSE                = 持久化 RunEvent 的浏览器传输协议
```

Worker 用受保护的数据库连接领取任务。浏览器只与 Auth 和 FastAPI 通信，不能直接读写业务表。任何跨 Git、Storage、Postgres 的发布都按 Saga/补偿执行，而不是承诺不存在的分布式事务。

## 2. 数据事实来源与实体

| 实体 | 最小关键字段 | 规则 |
| --- | --- | --- |
| `projects` | `id, owner_id, title, original_prompt, lifecycle_status, stable_version_id?, template_version, created_at, updated_at, archived_at?` | `stable_version_id` 是唯一稳定指针；生命周期不承载单次 Run 成败 |
| `build_contracts` | `id, project_id, version, content_json, status, created_by, approved_at?, superseded_at?` | 不原地覆盖；同项目按版本递增；同一时刻最多一个已批准版本 |
| `messages` | `id, project_id, run_id?, role, visible_content, summary?, created_at` | 只存用户可见文本/脱敏摘要；不存原始 Tool 输入输出 |
| `agent_runs` | `id, project_id, kind, request_id, contract_id?, base_version_id?, thread_id?, status, stage?, next_event_sequence, repair_attempts, cancel_requested_at?, lease_expires_at?, agent_config_version, model_id?, prompt_version, error_code?, error_message?` | 唯一 Run/队列记录；Planner 开始时尚无 Contract，批准前写入其 ID；保留可复现配置标识，不保存密钥或完整内部 Prompt |
| `run_events` | `id, run_id, sequence, type, schema_version, payload_json, created_at` | `(run_id, sequence)` 唯一；小型、脱敏、可回放事件 |
| `build_attempts` | `id, run_id, attempt_no, source_commit_sha?, status, diagnostics_ref?, build_log_ref?, runner_image_digest, duration_ms?, started_at, completed_at?` | 每个 Run 一次首次构建，最多一次 Repair 后重构建 |
| `project_versions` | `id, project_id, commit_sha, parent_version_id?, build_id, template_version, message, origin_kind, restored_from_version_id?, created_at` | 仅成功构建的真实 Commit 可入表；`(project_id, commit_sha)` 唯一 |
| `preview_artifacts` | `id, build_id, storage_key, integrity_hash, manifest_json, state, created_at` | 不保存易过期签名 URL；仅 `ready` Artifact 可成为稳定预览 |
| `agent_checkpoints` | `run_id, checkpoint_ref, updated_at` | 只映射 Deep Agents/LangGraph 恢复位置，不替代 Run 状态 |
| `exports` | `id, project_id, version_id, storage_key, expires_at, created_at` | 异步 ZIP 导出与过期清理 |

`template_version`、`agent_config_version`、`prompt_version`、`model_id` 和 `runner_image_digest` 是可复现性的最小审计字段：它们记录“用什么模板、Agent 配置和构建环境产生了版本”，但不记录 API Key、完整隐藏 Prompt 或未脱敏模型轨迹。

不保留 `commits.snapshot JSONB`、`agent_jobs`、持久化宿主机路径、持久化预览 URL，或以 `messages.tool_payload` 混存内部日志。

### 2.1 约束、索引与可见性

- `UNIQUE(project_id, request_id)`：客户端重复提交返回同一个 Run，而不是重复生成。
- 对流程型 Run 建部分唯一索引：同一项目最多一个 `queued/claimed/running/awaiting_approval/cancelling/promoting` Run；等待批准的 `initial` Run 已释放 Worker 租约，但仍占用项目的生成流程，不能并行创建第二个流程。
- `UNIQUE(run_id, sequence)`：同一 SSE 事件不会重复编号；`next_event_sequence` 与事件写入在同一短事务中递增。
- `CHECK(repair_attempts BETWEEN 0 AND 1)`，`UNIQUE(project_id, commit_sha)`，`UNIQUE(project_id, version)` 等规则落在数据库，不能只依赖 API。
- 应用表放在不公开给 Data API 的 schema；若有表因运维或查询需要暴露，必须显式启用 RLS 和所有权策略。API 的 ACL 是主控制，RLS 是防御纵深，而不是让浏览器绕过 API 的理由。

## 3. 状态机、队列与并发

### 3.1 项目、Contract、Run 三个不同维度

```text
Project lifecycle:
provisioning → draft → awaiting_approval → ready → archived

Build Contract:
draft → approved → superseded

Agent Run:
queued → claimed → running → awaiting_approval → queued → … → completed
                         ↘ cancelling → cancelled
                         ↘ failed

Run stage (status=running):
planning → generating → validating → typechecking → building
                         → repairing → validating  (最多一次)
                         → committing → promoting
```

页面不能把三者混成一个“项目状态”。项目列表显示 `lifecycle_status`，另附最近 Run 的状态/阶段（例如“ready · building”或“ready · failed”）；失败是最近一次 Run 的结果，不把项目生命周期写成 `failed`。

`initial` Run 的 Planner 成功后不继续 Builder：它创建草案 Contract、清空 Worker 租约并转为 `awaiting_approval`。用户批准后，API 把**同一 Run**重新置为 `queued`，其后从 `generating` 继续；这保证一个用户请求只有一个幂等 Run。`update`、`retry`、`restore` 都必须记录 `base_version_id` 并从已批准 Contract 开始；Repair 是同一 Run 的一个阶段，不是独立 Run 类型。

### 3.2 领取、租约与取消

```text
API command
    │ short transaction: validate ACL/Contract + insert agent_run(queued)
    v
Postgres queue ── FOR UPDATE SKIP LOCKED ──> Worker claims Run
    │                                           │ writes lease + event
    │                                           v
    └──────────────────────────────────────── Worker heartbeats/renews lease

cancel request → cancel_requested_at → Worker stops model/build child process
              → clean temp worktree → Run cancelled; stable pointer unchanged
```

- Worker 只在领取、续租、写事件、终态或提升稳定指针时使用短事务；模型调用与 Docker Build 期间不得持有数据库锁。
- 领取时使用 `FOR UPDATE SKIP LOCKED`；在项目级临界变更时使用事务级 `pg_advisory_xact_lock(project_id)`。租约到期的 Run 可由另一 Worker 依据 checkpoint 恢复；无法安全恢复时标记失败，绝不重复 Commit。
- 同一项目的写入型 Run 由部分唯一索引和项目锁双重保护。只读查询、日志与 SSE 不占项目锁。
- 取消是协作式：停止新 Tool 调用、取消模型请求、终止构建子进程、清理临时目录、保留诊断事件；候选 Commit/Artifact 不提升稳定指针，由清理任务审计处理。

## 4. 核心业务与数据流

### 4.1 总体闭环

```text
用户 ──JWT──> API ──ACL/command──> Postgres(agent_runs, events)
                                   │
                                   v
                        Worker claims a single Run
                                   │
  Git stable commit ──> temporary worktree ──> restricted Deep Agents
                                   │                    │
                                   │                    └─ sanitized progress
                                   v
                      deterministic Validator / Typecheck / Vite Build
                                   │
                       success     v       failure
                 Git candidate commit       Repair once or fail/cancel
                           │                         │
                           v                         └── stable version unchanged
               private Storage immutable Artifact
                           │
                           v
              short DB transaction: version + build + artifact + pointer
                           │
             API SSE <── persisted RunEvent ──> Workbench / owner preview
```

### 4.2 创建项目与批准 Contract

1. API 验证 JWT、创建项目 UUID 和独立 Git 仓库；仓库初始化失败时项目保持 `provisioning` 并写可诊断错误，后台补偿清理残留资源。
2. Planner Run 保存用户可见消息与脱敏事件，生成符合 Schema 的草案 Contract；成功后项目进入 `awaiting_approval`。
3. 编辑 Contract 必须创建新版本而非覆盖；批准时将旧批准版本标记 `superseded`，再将新版本置 `approved`，项目进入 `ready`。
4. `initial` Run 由需求创建并先完成 Planner；批准 API 将同一等待中的 `initial` Run 重新入队。`update`、`retry`、`restore` 必须使用已批准 Contract。进入 `generating` 前，API 冻结该 Run 的 Contract、基线版本、模板与 Agent 配置版本。

### 4.3 生成、构建与稳定提升 Saga

```text
1. claim Run / pin base version
2. create isolated worktree from base Git Commit
3. Builder writes allowlisted source paths
4. deterministic validation + typecheck + production build
5. [optional] Repair once → repeat step 4
6. create candidate Git Commit (tagged with run_id)
7. upload immutable Artifact (key includes run_id + hash)
8. one DB transaction:
   insert build_attempt/project_version/preview_artifact
   update projects.stable_version_id
   mark Run completed and append final event
9. cleanup temporary worktree

Any failure before step 8: do not update stable_version_id or stable preview.
Cleanup reconciles orphan candidate commits/Artifacts by run_id and retention policy.
```

第 8 步是唯一改变用户“当前版本”的事务边界。Git Commit 与 Storage 上传成功但数据库提升失败时，它们是可审计孤儿，不是新稳定版本。恢复版本同样从目标 `project_version` 建 worktree、重新校验和构建，并创建新的 `restore` Commit；禁止改写历史 Commit。

### 4.4 预览、导出与访客

- 稳定 Artifact 位于私有 bucket。API 只为项目所有者（含受控匿名访客会话）签发短时预览票据；Preview Gateway 验证后只读服务对应 Artifact。
- 平台的公开链接不等于项目公开分享：访客模式使用隔离匿名身份及临时项目，不能猜测或访问其他用户项目、事件、日志、版本或预览。
- Export 由异步 Run/任务生成私有 ZIP；下载通过短时授权链接，过期后只删除导出对象，不影响 Git 与稳定 Artifact。

## 5. SSE 产品协议

### 5.1 连接与恢复

```http
GET /v1/projects/{projectId}/runs/{runId}/events HTTP/1.1
Authorization: Bearer <supabase-access-token>
Accept: text/event-stream
Last-Event-ID: 42
```

API 先完成项目所有权检查，再查询 `sequence > 42` 的持久化事件并按序回放，随后保持连接推送新增事件。服务响应包含：

```http
HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache, no-transform
Connection: keep-alive
X-Accel-Buffering: no
```

- `id` 始终为 Run 内单调递增 `sequence`，不是数据库 UUID；客户端把最后收到的 `id` 作为 `Last-Event-ID`。
- 连接中断使用指数退避重连；每次重连先回放缺口，再订阅新增事件。页面刷新先读取 Run 快照，再用最后序号建立 SSE。
- 事件保留期过后，API 返回 `410 event_history_expired` 和可安全展示的 Run 快照 URL；客户端刷新快照并从当前最大序号重新连接，不猜测遗漏的进度。
- SSE 只能携带产品级、脱敏字段。原始 Prompt、模型隐藏推理、密钥、完整 Tool 输入输出和宿主机路径不得出现在事件中。

### 5.2 事件信封与类型

所有事件使用命名空间点号，避免与内部字段混淆；前后端从 `contracts/sse/v1.schema.json` 生成/校验类型。

```text
run.queued | run.claimed | stage.started | contract.ready | run.awaiting_approval
agent.message | agent.file.changed | build.started | build.diagnostic | build.completed
repair.started | version.promoted | run.completed | run.failed | run.cancelled
```

```text
id: 43
event: stage.started
data: {"schemaVersion":1,"runId":"run_01","sequence":43,"at":"2026-07-19T10:03:14Z","payload":{"stage":"building","message":"正在执行生产构建"}}

id: 44
event: build.diagnostic
data: {"schemaVersion":1,"runId":"run_01","sequence":44,"at":"2026-07-19T10:03:20Z","payload":{"attempt":1,"severity":"error","code":"TS2322","file":"src/App.tsx","line":18,"message":"类型不兼容","retryable":true}}

id: 45
event: version.promoted
data: {"schemaVersion":1,"runId":"run_01","sequence":45,"at":"2026-07-19T10:04:03Z","payload":{"versionId":"ver_01","buildId":"build_01","previewState":"ready"}}
```

`run.claimed` 不暴露 `workerId`；如需显示，仅提供对用户无敏感意义的阶段消息和租约状态。大日志上传 Storage 后，事件只带 `diagnosticsRef` 或受控日志查询入口，不嵌入整份日志。

### 5.3 前端状态数据

前端以 TanStack Query 保存产品级 Run 投影，以项目级 Zustand Store 保存 `connectionState` 与 `lastAppliedSequence` 等连接 UI；两者不得重复保存完整 Run、文件、版本或事件日志。具体目录和 Store 边界见 [目录与前端模块设计](./4-目录与前端模块设计.md)。前端不以“收到某条文本”推断最终成功：

```ts
type RunProjection = {
  runId: string;
  status: 'queued' | 'claimed' | 'running' | 'awaiting_approval' | 'cancelling' | 'completed' | 'failed' | 'cancelled';
  stage?: 'planning' | 'generating' | 'validating' | 'typechecking' | 'building' | 'repairing' | 'committing' | 'promoting';
  lastSequence: number;
  repairAttempts: number;
  cancelRequestedAt?: string;
  error?: { code: string; message: string; retryable: boolean };
  stableVersionId?: string;
};
```

Reducer 只接受 `sequence > lastSequence` 的事件；同序号忽略、缺号则重连并重新获取快照。浏览器传输使用封装的 `@microsoft/fetch-event-source`，使 Bearer Header、Abort、410 恢复和可控退避不泄漏到 UI。`version.promoted` 才刷新文件树、版本列表和预览票据；`run.failed`/`run.cancelled` 保留现有稳定版本并展示诊断与重试操作。API 快照字段与该投影同源，避免刷新后 UI 状态倒退。

## 6. 数据保留、可观测性与验收

- 用户消息、Contract、成功版本和稳定 Artifact 随项目保存；项目先归档，删除按明示保留策略和审计流程执行。
- 原始运行日志默认保存 30 天；脱敏 `run_events` 默认保存 90 天；匿名访客项目、候选 Commit、孤儿 Artifact 和过期导出由定期任务清理并保留审计条目。
- LangSmith 可用于开发期 tracing/Eval；线上产品 Run 状态、错误码、审查证据和 Eval 结果必须能脱离 LangSmith 查看。

必须通过的系统级验收：

1. 同项目重复 `request_id` 只得到同一个 Run；并发请求最多一个写入型 Run。
2. 多 Worker 领取时，同一 Run 不会被执行两次；租约过期可安全恢复或失败，不产生重复 Commit。
3. 模型、校验、Git、上传或数据库任一失败，稳定版本、稳定预览均不变。
4. SSE 断线后 sequence 连续回放；过期历史走 410+快照恢复；事件不泄露内部 Tool 轨迹与密钥。
5. 用户/访客之间无法跨项目读取项目、事件、日志、版本、Artifact、预览票据或导出。
6. Repair 最多一次；Restore 创建新 Commit/新版本而非修改历史。
7. 三个 Eval 场景完成首次生成、核心交互、刷新恢复、增量修改、失败保护与恢复，且有可复查记录。
