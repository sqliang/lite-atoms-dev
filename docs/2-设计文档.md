# Lite Atoms Dev 架构总览

> 版本：3.0  
> 本文定义服务边界、技术选型和部署方式；运行状态、存储模型、数据流与 SSE 契约见 [运行时与数据设计](./3-运行时与数据设计.md)。产品范围以 [PRD](./1-PRD.md) 为准。

## 1. 阅读顺序与设计原则

| 文档 | 回答的问题 |
| --- | --- |
| [1-PRD.md](./1-PRD.md) | 用户能完成什么、如何验收？ |
| 本文 | 哪些服务负责什么，边界如何保证？ |
| [3-运行时与数据设计.md](./3-运行时与数据设计.md) | 状态如何流转、数据如何落库、SSE 如何恢复？ |
| [4-目录与前端模块设计.md](./4-目录与前端模块设计.md) | 前后端如何落入目录，前端状态如何分层？ |

不可变原则：

- Git 是生成源码及其真实版本历史的唯一事实来源；
- PostgreSQL 是项目、Run、版本索引和产品状态的事实来源；Storage 只保存不可变 Artifact、日志和导出文件；
- 只有校验、构建、Git 提交、Artifact 上传和数据库提升均成功后，稳定版本才可变更；
- Agent 只在受限 Tool 与临时 worktree 内推理和修改源码，不能取得平台密钥、Docker Socket、Git 凭据或数据库权限；
- SSE 传递持久化且脱敏的产品事件，不把框架内部事件格式直接暴露给浏览器。

## 2. 系统架构

```text
                                  Internet
                                     |
          +--------------------------+--------------------------+
          |                          |                          |
          v                          v                          v
 +------------------+       +------------------+       +------------------+
 | app.<domain>     | JWT   | api.<domain>     | ticket| preview.<domain> |
 | React Workbench  |------>| FastAPI API      |------>| Preview Gateway  |
 | Contract / Files |<------| REST / SSE / ACL |       | stable dist only |
 +------------------+  SSE  +---------+--------+       +--------+---------+
                                         |                     |
                        trusted DB/Store |                     | private read
                                         v                     v
                     +-----------------------------------------------+
                     | Supabase: Auth | PostgreSQL | private Storage |
                     +------------------------+----------------------+
                                              ^
                                              | short DB transactions
                     +------------------------+----------------------+
                     | Worker: queue | state machine | publish        |
                     +-----------+--------------+--------------------+
                                 |              |              |
                                 v              v              v
                       +---------------+ +-------------+ +----------------+
                       | Deep Agents   | | project Git | | Build Runner   |
                       | restricted    | | source truth| | rootless/no net|
                       | Planner etc.  | | temp trees  | | resource limits|
                       +---------------+ +-------------+ +----------------+
```

控制面（Workbench、API、PostgreSQL）管理身份、产品状态和可见性；执行面（Worker、受限 Agent、Build Runner）处理长任务和不可信源码。生成应用与平台使用不同 Origin，不能读取平台 Cookie 或调用平台内部接口。

### 2.1 服务职责

| 服务 | 负责 | 不负责 |
| --- | --- | --- |
| Frontend | 登录、Contract 编辑、工作台、SSE 消费、文件/日志/版本展示 | 构建、Git、服务端写入与密钥保存 |
| FastAPI API | JWT 校验、所有权检查、命令入口、SSE 回放、预览票据 | 长模型调用、长构建任务 |
| Worker | 领取 Run、Agent 调用、确定性校验、Git、构建、产物提升、事件写入 | 向浏览器泄露主机路径或服务凭据 |
| Build Runner | 临时 worktree 中执行 TypeScript 与 Vite build | 网络、平台数据库、Git 凭据、宿主机敏感目录 |
| Preview Gateway | 独立 Origin 下只读提供成功 `dist/` | 平台 API、写入操作或平台登录会话 |
| Supabase | Auth、托管 Postgres、私有 Storage | Agent、Git、Vite 或任意代码执行 |

Preview Gateway 可以在验证短时预览票据后设置**仅绑定项目与 Artifact 的短时预览 Cookie**；它不是平台用户会话，不能扩大为访问其他 API 或项目的凭据。

## 3. 技术选型

| 层级 | 选型 | 理由与约束 |
| --- | --- | --- |
| 平台前端 | React + TypeScript + Vite | 工作台交互与类型约束；生成应用也固定在此受限模板内 |
| UI / 路由 | Tailwind CSS + shadcn/ui + React Router | 快速形成一致工作台，不增加独立 UI 运行时 |
| 前端服务端状态 | TanStack Query | 缓存、失效、刷新恢复及 SSE 对服务端投影的归并；不复制到全局 Store |
| 前端本地状态 | Zustand（项目级 vanilla store） | 标签、布局、选中项和 SSE 连接 UI 等跨面板短生命周期状态；Store 按 `projectId` 隔离 |
| 浏览器 SSE 传输 | `@microsoft/fetch-event-source` 的本地适配层 | 支持 Bearer Header、Abort、状态码分类与可控重试；第三方库不泄漏到 feature 层 |
| API / Worker | Python 3.11+ + FastAPI | REST、SSE、鉴权边界和独立长任务 Worker |
| Agent Harness | `deepagents` | Agent 主入口，提供计划、上下文、文件 Tool、流式事件与检查点 |
| Agent 扩展 | LangChain Core + 模型厂商 SDK + Pydantic | 模型适配、自定义 Tool、结构化输出与 Schema 校验 |
| Agent Runtime | LangGraph（Deep Agents 内置） | 使用其运行时与检查点，不另写一套平行编排器 |
| 数据与身份 | Supabase Auth + PostgreSQL + Storage | 身份、关系数据、RLS 防御与私有对象存储 |
| 源码与版本 | Git | 真实 Commit、Diff、恢复、导出 |
| 隔离构建 | Rootless Docker | 非 root、无网络、短生命周期、CPU/内存/磁盘/超时限制 |

Deep Agents、LangChain 和 LangGraph 不是三套并列工作流：Deep Agents 是入口；LangChain 只用于模型、Tool 与结构化输出；LangGraph 通过 Deep Agents 间接使用。

参考：[Deep Agents 概览](https://docs.langchain.com/oss/python/deepagents/overview)、[Deep Agents 沙箱说明](https://docs.langchain.com/oss/python/deepagents/sandboxes)。

## 4. Agent 与确定性执行边界

```text
用户需求 → Planner（只读需求） → Build Contract JSON → 用户批准
                                                    ↓
当前稳定源码 → Builder（受限写入） → Validator → Typecheck/Build
                                                     ↓失败且可修复一次
                                              Repair（受限写入）
                                                     ↓成功
                               Worker: Git Commit → Artifact → DB 提升
```

| 角色/组件 | 可访问输入 | 权限 | 输出 |
| --- | --- | --- | --- |
| Planner | 需求、项目摘要 | 无源码、Git、Shell、网络或数据库 | 符合 Schema 的 Build Contract |
| Builder | 已批准 Contract、稳定源码摘要 | 只读稳定源码；仅可写白名单 `src/` 子目录 | 变更文件清单与摘要 |
| Repair | 构建诊断、Contract、临时源码 | 同 Builder；最多一次 | 定向修复 |
| Worker 确定性步骤 | 完整临时 worktree | 校验、构建、Git、上传和提升 | 版本、Artifact、Run 终态 |

默认禁用 Deep Agents 的通用 Shell、`delete`、泛化子 Agent、MCP、网络和数据库 Tool。Validator、Build、Git Commit、Artifact 上传、稳定指针更新不作为模型可自由调用的 Tool。

## 5. 代码库模块边界

```text
lite-atoms-dev/
├── app/                          # 可部署应用：frontend、backend、preview-gateway
├── contracts/                    # API、SSE、Build Contract 的版本化 Schema
├── templates/                    # 受控 React/Vite/Tailwind 生成模板
├── supabase/                     # migration、角色与 RLS/Storage 策略
├── infra/                        # Compose、镜像、反向代理、资源限制
├── evals/                        # 固定 Prompt、验收步骤与结果格式
├── docs/                         # task、PRD、架构、运行时和模块设计
├── scripts/                      # 开发、迁移、清理和运维辅助脚本
├── compose.yaml
└── README.md
```

后端建议按领域而非按页面拆分：`api/`（路由与鉴权）、`domain/`（Project/Run/Version 状态规则）、`agents/`（Deep Agents 与受限 Tool）、`execution/`（worktree/validator/build/git）、`persistence/`（仓储与事务）、`streaming/`（SSE）和 `security/`（ACL、票据、脱敏）。前后端不得复制契约：`contracts/` 是 API、SSE、Contract Schema 的唯一版本化来源。完整树、既有前端的迁移边界与状态职责见 [目录与前端模块设计](./4-目录与前端模块设计.md)。

## 6. 部署、评审访问与安全基线

- 首版可用单台云主机 Docker Compose 部署 `frontend`、`api`、`worker`、`preview-gateway`；Supabase 是外部托管依赖。该部署适合作业规模，服务边界仍保留，未来可独立扩容 Worker。
- 浏览器只使用 Supabase Auth 公钥和 FastAPI；不直连业务表、Storage 私有 bucket 或 Worker。
- API 先做所有权检查；对可能暴露的表显式启用 RLS 作为第二道防线。服务端使用最小权限连接，不把 Service Role Key 发给浏览器。
- 评审可从公开平台链接进入受控访客模式：以匿名 Auth 会话创建或进入**仅属于该访客的临时项目**，预览仍仅限该会话；不提供任意项目的公开分享链接。临时访客项目按明确保留策略归档/清理。
- Build Runner 使用固定镜像 digest、非 root 用户、只挂载临时 worktree、禁网与资源上限；Preview 使用独立 Origin、CSP、iframe sandbox 和受限网络策略。生成应用可使用自身 Origin 的 `localStorage`/`sessionStorage` 保存无敏感 UI 状态，但校验器仍禁止 `document.cookie`、动态执行和动态 import；平台会话、票据与业务数据不可被预览应用读取。

## 7. 交付节奏与评审证据

P0/P1 仍表示开发顺序，不改变 PRD 中的完整交付范围。为避免“页面完成但链路不真实”，实施按证据闭环推进：

1. 认证、项目、Contract、持久化与所有权隔离；
2. 受限生成 → 真构建 → 稳定预览 → SSE 可恢复；
3. 增量修改、失败保护、真实 Git、版本 Diff/恢复/导出；
4. 三个 Eval 场景、无痕访问、README、公开代码与部署证据。

每一阶段以可运行演示和对应验收记录为完成条件，不以 UI 截图或模拟进度代替。

## 8. 关键决策与未扩展范围

| 决策 | 目的 |
| --- | --- |
| 不引入 Redis/Celery | 本次以 Postgres 租约队列满足规模与可审计性，减少运行依赖 |
| 不使用 Supabase Realtime 传递 Run 状态 | SSE 支持顺序号、重放和浏览器协议控制 |
| 不用 JSONB 快照模拟 Commit | 真实 Git 才能支持可信 Diff、恢复与导出 |
| 不让 Agent 执行任意命令 | 安全性来自 Tool/容器边界，不依赖 Prompt 自觉 |
| 不开放项目公开分享 | 作业公开的是平台入口；访客模式隔离其项目，避免数据与预览越权 |

运行时状态机、表结构、Saga、失败补偿、SSE 样例、数据保留与验收细则均见 [运行时与数据设计](./3-运行时与数据设计.md)。
