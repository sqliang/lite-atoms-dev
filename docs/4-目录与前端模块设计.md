# Lite Atoms Dev 目录与前端模块设计

> 版本：1.0  
> 本文补充 [架构总览](./2-设计文档.md)：将服务边界落为整体目录，并给出当前前端原型到目标结构的渐进迁移方案。产品范围以 [PRD](./1-PRD.md) 为准，Run/SSE 产品契约以 [运行时与数据设计](./3-运行时与数据设计.md) 为准。

## 1. 当前实现的客观评估

现有前端已经具备 React/Vite、React Router、Supabase 邮箱登录/注册、项目首页和工作台 UI 原型，`WorkspaceContext` 已实现文件树与编辑器标签交互。这部分应保留视觉和交互资产，不能为了“目录整洁”而整页重写。

但它仍是原型实现，存在必须明确隔离的边界：

- 页面直接读写特定 Supabase 业务表，和 PRD 的 FastAPI 统一鉴权/API 边界冲突；Supabase 浏览器客户端最终只保留 Auth。
- 首页自动写入 mock 项目的 RPC、工作台内嵌的文件/消息/版本 Demo 数据，会污染真实项目数据并混淆验收。
- `WorkspaceContext` 同时保存文件、标签和操作方法；后续叠加 Run、SSE、版本和预览会形成难以维护的全局 Context。
- `blog/`、`prerender/`、MetaGPT SDK 是原型来源，不属于 Task 或 PRD 的 AI 开发工作台范围；不应成为正式运行时依赖。

结论：采用**渐进迁移，不保留正式 legacy 层**。先保持已有 UI，逐个迁移数据入口和状态所有权；完成迁移后删除原型模块与无用依赖。

### 2.1 模块依赖规则

- `pages` 只能组合 `features`，不得直接调用 Supabase、FastAPI 或 Store。
- `features` 可调用 `shared`，但 feature 之间通过公开 hook/DTO 交互，禁止深层相互 import。
- `shared/api` 是浏览器调用 FastAPI 的唯一入口；`shared/auth` 是浏览器调用 Supabase Auth 的唯一入口。
- `backend/api` 不直接读写 Git、Docker 或 SQL；它调用 `application`。`application` 调用 `domain` 和接口，具体实现由 `infrastructure`/`execution` 提供。
- `contracts` 只存版本化协议，不存运行逻辑；Build Contract 和 SSE 事件均以此处 Schema 为准。

## 3. 前端状态与数据访问

### 3.1 状态所有权

| 状态 | 唯一归属 | 说明 |
| --- | --- | --- |
| 登录用户、Session | Supabase Auth + `AuthProvider` | 低频外部认证订阅，保留现有模式 |
| Project、Contract、消息、Run、文件、Build、Version、预览票据 | TanStack Query | 远端事实来源；负责缓存、失效、刷新恢复 |
| SSE 事件归并 | `features/runs` reducer + Query cache | 事件更新服务端状态投影，不能另存完整事件副本 |
| SSE 连接状态、最后已应用 sequence | Zustand `streamUiSlice` | 只服务连接提示、重连和去重 UI |
| 标签、激活文件、树折叠、面板布局、当前视图/版本 | Zustand `workspaceStore` | 项目级、跨面板、短生命周期状态 |
| 输入草稿、弹窗、滚动、图片缩放 | 组件 `useState` | 单组件状态，不上移 |

Zustand 不替代 React Query：Query 管理异步服务器状态；Zustand 管理客户端工作台协调。禁止在两者中同时保存完整 Project、Run、文件内容或版本实体。

### 3.2 项目级 Zustand Store

每个 `WorkspacePage(projectId)` 创建一个 Zustand vanilla store，并通过 `WorkspaceStoreProvider` 以 Context 注入。Store 生命周期随页面卸载而结束；切换项目必须创建新实例，不能复用旧项目的标签、选中版本或 SSE 序号。

```text
workspace-store
├── layoutSlice       # 面板尺寸、文件树展开/折叠、当前视图
├── editorSlice       # 打开标签、激活标签、文件选择
├── navigationSlice   # 当前 Activity/Files/Build/Diff/Preview 面板
└── streamUiSlice     # connecting/open/reconnecting/error、lastAppliedSequence
```

第一版不使用 `persist`。刷新后的项目、文件、Run 和预览全部从 API 快照恢复；以后如需持久化，仅允许将无敏感的布局偏好写入 `sessionStorage`，按 `projectId` 和 schema version 隔离。

### 3.3 Mock 与迁移边界

- `mocks/` 只提供开发 fixture，由显式 `VITE_DATA_MODE=mock` 启用；默认/生产构建不得导入 fixture，更不得调用 RPC 自动插入演示项目。
- 当前直接 Supabase 项目表调用先收敛为 `features/projects/api` 的临时适配器，保持已有 UI 可用；FastAPI Project API 完成后替换该适配器，页面无需再次重写。
- 当前 `WorkspaceContext` 迁移为 `workspaceStore` 后，文件/标签交互 API（如 `openTab`、`closeTab`、`setActiveTab`）保持语义不变，降低 UI 改动风险。
- `blog/`、`prerender/`、MetaGPT 模块和相关依赖在工作台 Feature/API 替换完成后删除；若原型截图需保留，移至 `docs/assets/`。

## 4. SSE 浏览器传输设计

### 4.1 选型与封装

采用固定版本 `@microsoft/fetch-event-source@2.0.1`，但只允许 `features/runs/api/sse-transport.ts` 直接 import。原生 `EventSource` 不能传 `Authorization` header，也不能细分 401/403/410；手写 Fetch 则需自行维护 SSE 帧解析、重连和 Page Visibility。该库支持标准 Event Stream、Fetch headers、AbortController、响应检查及重试控制，正好覆盖本项目协议。

该包正式 release 较旧，故通过本地适配层隔离：精确锁定版本、用协议测试覆盖行为、禁止 feature UI 依赖包类型。未来替换为自研 Fetch parser 不影响 `useRunStream` 或 UI。

唯一接口：

```ts
connectRunEvents(input: {
  projectId: string;
  runId: string;
  lastEventId?: number;
  signal: AbortSignal;
  onEvent: (event: RunEvent) => void;
  onConnectionState: (state: SseConnectionState) => void;
}): Promise<void>
```

### 4.2 连接、认证与恢复规则

1. 以 `GET /v1/projects/{projectId}/runs/{runId}/events` 发起连接，传 `Authorization: Bearer <fresh Supabase token>`、`Accept: text/event-stream` 与 `Last-Event-ID`；Token 禁止出现在 URL、日志或 Zustand。
2. `onopen` 必须验证 `200` 和 `Content-Type: text/event-stream`。`401` 只尝试一次刷新 Session 后重连；`403`/`404` 终止；`410 event_history_expired` 先获取 Run 快照和最大 sequence，再建新连接。
3. 网络错误、`429`、`5xx` 使用含抖动的退避：1s、2s、4s、8s，最大 10s。用户取消、路由切换和 Run 终态通过 `AbortController` 立即关闭，不再重连。
4. 页面隐藏可关闭连接以节约资源；重新可见后使用最后成功应用的 event id 回放。服务端持久化事件，因此页面隐藏不丢产品状态。
5. 每条消息同时校验 SSE `id` 与 JSON payload 的 `sequence`/`runId`/`schemaVersion`。重复序号忽略；缺号、非法 JSON 或 Schema 不符时放弃本地推断，重新取 Run 快照。
6. reducer 只更新 Query cache；`version.promoted` 失效文件树、版本列表和预览票据。Zustand 仅更新连接 UI 与最后已应用 sequence。

后端/网关必须继续输出 `Cache-Control: no-cache, no-transform`、`X-Accel-Buffering: no`、心跳注释和单调递增 SSE `id`；CORS allowlist 允许平台 Origin、`Authorization` 与 `Last-Event-ID`。

## 5. 迁移顺序与验证

1. 建立目录与 import alias；保持 `AuthProvider`、现有页面和 shadcn UI 外观不变。
2. 将 Workspace 的标签/布局状态迁到项目级 Zustand Store，验证项目切换会销毁旧 Store。
3. 将 mock 数据移入 `mocks/`，移除自动写库；将页面 Supabase 项目调用收敛到 feature API 适配器。
4. 引入 FastAPI Project/Run API 后，替换临时适配器；实现 Query hooks 和 `useRunStream`。
5. 完成 SSE transport，验证 Authorization、Abort、401 单次刷新、403/404 停止、410 快照恢复、429/5xx 退避、重复/缺失 sequence。
6. 移除博客/预渲染/MetaGPT 原型模块和未使用依赖；完成前端 build/lint、登录、创建项目、文件切换、刷新恢复和断线重连验收。
