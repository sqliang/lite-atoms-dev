# Lite Atoms Dev — 实现说明与扩展计划

> 版本：1.1（2026-07-19 更新）  
> 定位：面向评审的简要交付说明，覆盖实现思路、关键取舍、完成度、创新点与后续扩展优先级。  
> 1.1 变更：同步 Plan/Build 双模式、代码引用上下文、版本只读回溯、消息持久化、本地 Postgres 拓扑等已实现能力，并更新验证结果。

---

## 核心判断

**Lite Atoms Dev 不是一个“界面演示型”的 AI Coding Demo，而是一个以“可信生成闭环”为核心设计的工程原型。**

在 6–8 小时的 Demo 约束下，我们优先保证：

1. **用户主链路真实可运行**——从自然语言需求到可交互预览的完整闭环；
2. **生成过程可解释、可失败、可恢复**——SSE 实时事件、持久化状态、真实 Git 版本；
3. **安全边界不依赖模型自觉**——Agent 只能操作受限文件 Tool，校验/构建/提交/提升由确定性 Worker 执行。

取舍原则非常明确：**先让“计划 → 审批 → 生成 → 校验 → 构建 → 提交 → 预览”这一最小可信闭环真正跑通，再补充周边能力。** 这一判断直接决定了技术选型、模块边界和当前完成度。

## 一、实现思路与关键取舍

### 1.1 总体架构：控制面与执行面分离

```text
Internet
   │
   ├─► app.<domain>      React 工作台（仅 Supabase Auth + FastAPI）
   ├─► api.<domain>      FastAPI 控制面（鉴权、命令、SSE 回放、预览票据）
   └─► preview.<domain>  隔离预览网关（仅服务已构建 dist）
           │
           ▼
    Supabase Auth（身份）/ Postgres（产品状态）/ Storage（产物）
           ▲
    Worker ──► Deep Agents（Planner / Builder / Repair）
           │
           ▼
    临时 worktree ──► AST 校验 ──► TypeScript/Vite Build ──► Git Commit ──► Artifact
```

> 本地运行拓扑：compose 编排的 `db` 服务（postgres:16）承载产品状态，Supabase 仅承担 Auth（JWKS 本地验签）；Supabase Postgres/Storage 仍是生产路径，代码通过 `DATABASE_URL` 一项配置即可切换。

**核心边界：**

| 层级 | 负责 | 不负责 |
| --- | --- | --- |
| 前端工作台 | 登录、Contract 编辑、SSE 消费、文件/日志/版本展示 | 构建、Git、服务端写入、密钥保存 |
| FastAPI API | JWT 校验、所有权检查、命令入口、SSE 回放、预览票据 | 长模型调用、长构建任务 |
| Worker | 领取 Run、Agent 调用、确定性校验、Git、构建、产物提升、事件写入 | 向浏览器泄露主机路径或服务凭据 |
| Build Runner | 临时 worktree 中执行 TypeScript 与 Vite Build | 网络、平台数据库、Git 凭据、宿主机敏感目录 |
| Preview Gateway | 独立 Origin 下只读提供成功 `dist/` | 平台 API、写入、平台登录会话 |
| Supabase | Auth、托管 Postgres、私有 Storage | Agent、Git、Vite、任意代码执行 |

### 1.2 事实来源设计

| 数据 | 事实来源 | 原因 |
| --- | --- | --- |
| 生成源码与真实版本历史 | Git | 支持可信 Diff、恢复、导出，拒绝 JSONB 快照模拟 Commit |
| 项目、Run、版本索引、事件 | Postgres（本地为 compose `db` 服务，生产为 Supabase Postgres） | 关系查询、事务、队列租约、访问控制 |
| 不可变 Artifact、长日志、导出 | Supabase Storage / 本地卷 | 只保存成功构建产物，失败产物不进入稳定指针 |

### 1.3 Agent 与确定性执行边界

Agent（Deep Agents + LangChain/LangGraph）只拥有三种能力：

- `read_source`：读取临时 worktree 内源文件；
- `list_sources`：列出可编辑源文件；
- `write_source`：在白名单路径（`src/`、`public/`）内写入 bounded 文件。

**校验、TypeScript 检查、Vite Production Build、Git Commit、Artifact 复制、数据库稳定指针更新，全部在 Worker 中以确定性代码完成，不作为 Agent Tool 暴露。** 这样即使模型产生错误输出，也只会停留在临时 worktree，不会污染稳定版本。

### 1.4 SSE 与可解释进度

前端通过 `@microsoft/fetch-event-source` 的本地适配层消费 SSE。每个事件持久化到 `run_events` 表并带单调递增 `sequence`。连接中断后通过 `Last-Event-ID` 回放，过期后走 `410` 快照恢复。**禁止使用定时器伪造进度。**

### 1.5 前端状态分层

| 状态 | 归属 | 说明 |
| --- | --- | --- |
| 用户/Session | Supabase Auth + `AuthProvider` | 低频外部认证订阅 |
| Project、Contract、Run、文件、Build、Version、预览票据 | TanStack Query | 远端事实来源；负责缓存、失效、刷新恢复 |
| SSE 连接状态、最后已应用 sequence | Zustand `streamUiSlice` | 只服务连接提示、重连和去重 UI |
| 标签、布局、选中项 | Zustand `workspaceStore` | 项目级、跨面板、短生命周期状态 |
| 输入草稿、弹窗、滚动 | 组件 `useState` | 单组件状态，不上移 |

**禁止在 React Query 和 Zustand 中同时保存完整 Project/Run/文件实体。**

### 1.6 关键技术选型与取舍

| 领域 | 选型 | 理由 |
| --- | --- | --- |
| 平台前端 | React + TypeScript + Vite + Tailwind + shadcn/ui | 与生成应用模板一致，减少心智负担 |
| 前端服务端状态 | TanStack Query | 缓存、失效、刷新恢复及 SSE 对服务端投影的归并 |
| 前端本地状态 | Zustand（项目级 vanilla store） | 轻量、按 `projectId` 隔离、无持久化陷阱 |
| 浏览器 SSE | `@microsoft/fetch-event-source` 本地适配层 | 支持 Bearer Header、Abort、状态码分类、可控重试 |
| API / Worker | Python 3.11+ + FastAPI | REST、SSE、鉴权边界、独立长任务 Worker |
| Agent Harness | `deepagents` + LangChain Core + LangGraph | 提供计划、上下文、文件 Tool、流式事件与检查点 |
| 数据与身份 | Supabase Auth + PostgreSQL（本地 compose 容器 / 生产 Supabase）+ Storage | 身份（JWKS 本地验签）、关系数据、RLS 防御、私有对象存储 |
| 隔离构建 | Rootless Docker（Build Runner） | 非 root、无网络、短生命周期、资源限制 |

### 1.7 关键取舍

| 决策 | 取舍说明 |
| --- | --- |
| **不引入 Redis/Celery** | 以 Postgres 租约队列满足 Demo 规模与可审计性，减少运行依赖；未来 Worker 可独立扩容。 |
| **不使用 Supabase Realtime 传递 Run 状态** | SSE 支持顺序号、重放和浏览器协议控制，更适合长事件流。 |
| **不用 JSONB 快照模拟 Commit** | 真实 Git 才能支持可信 Diff、恢复与导出；数据库只保存版本索引。 |
| **不让 Agent 执行任意命令** | 安全来自 Tool/容器边界，不依赖 Prompt 自觉。 |
| **不开放项目公开分享** | 公开的是平台入口；访客模式隔离其临时项目，避免越权。 |
| **Plan/Build 双模式** | Build 模式 Planner 产出合法 Contract 后自动批准直接生成，保证 Demo 流畅；Plan 模式按 Run 暂停在 `awaiting_approval`，用户在会话中审阅、编辑（产生新 Contract 版本）并批准后才进入生成。 |
| **P0 预览产物存本地卷** | 数据库模型已预留 Supabase Storage 字段；本地卷足够验证稳定提升语义，Storage 适配作为 P1。 |

## 三、关键创新点

1. **“先规划、后生成、可构建验证、可回滚”的可信工作流**
   把 AI Coding 从“聊天直接改代码”提升为带有明确审批门和构建验证的受控流程；Contract 是产品边界，批准后才进入生成。

2. **Agent 能力与平台权力的严格分离**
   Agent 只拥有受限文件 Tool；校验、构建、Git、数据库提升等关键动作由确定性 Worker 执行。模型错误不会直接变成稳定版本。

3. **真实 Git 作为源码与版本的事实来源**
   每次成功构建都是真实 Commit；失败、取消、修复失败均不移动 `stable_version_id`，旧版本不会丢失。

4. **持久化 SSE 作为可解释进度协议**
   事件带单调 `sequence`、支持断线续传与 410 快照恢复，前端不依赖假进度条或文本推断状态。

5. **清晰的前端状态所有权**
   React Query 管理服务器事实，Zustand 管理 UI 协调，组件状态管理本地交互；避免一个全局 Context 承载所有职责。

## 四、如果继续投入，如何扩展及优先级

### 4.1 核心判断：Agent 优化是第一优先级

当前 Lite Atoms Dev 的 P0 已经证明：**“计划 → 审批 → 生成 → 校验 → 构建 → 提交 → 预览”这个可信闭环可以跑通。** 但系统真正的生产力瓶颈和价值上限，取决于 Agent 的生成质量与上下文利用效率。

因此，如果继续投入，我会把 **Agent 能力升级** 作为第一优先级，而不是先补大量平台周边功能。原因是：

- 用户最终感知到的价值几乎全部由 Agent 输出决定；
- 上下文越精准，生成越稳定，幻觉和 Repair 次数越少；
- 多 Agent 分角色协作是工程规模化必经之路，能让复杂需求拆解更合理；
- 平台周边能力（Diff、导出、访客模式）是“兜底和交付必需品”，Agent 优化才是“核心价值放大器”。

### 4.2 Agent 优化的四个方向

#### 方向一：从单 Agent 执行升级为多 Agent 协作

当前实现本质上是 Worker 内顺序调用 Planner / Builder / Repair 三个 Prompt+Tool 组合。下一步应拆分为真正独立的 Agent 角色，由 LangGraph 编排：

| Agent 角色 | 职责 | 可用 Tool / 能力 | 输出 |
| --- | --- | --- | --- |
| **Planner** | 需求分析与 Contract 生成 | 只读需求、项目摘要 | Build Contract JSON |
| **Architect** | 技术方案与组件拆分 | 读取稳定源码、读取 Contract | 组件图、状态方案、文件拆分建议 |
| **Builder** | 具体代码实现 | 受限文件 Tool（读/写 `src/`、`public/`） | 变更文件清单与摘要 |
| **Reviewer** | 代码审查与潜在问题发现 | 读取源码、读取 Contract、读取 Build 诊断 | 审查意见或“通过” |
| **Repair** | 基于诊断定向修复 | 受限文件 Tool | 修复后的文件变更 |
| **Test Writer** | 生成验收测试/交互用例 | 读取源码、读取 Contract | 测试脚本或验收步骤 |

编排方式示例：

```text
需求 → Planner → Contract
             ↓
        Architect（可选）→ 技术方案
             ↓
        Builder 生成源码
             ↓
        Reviewer 审查 ──→ 不通过 → 回到 Builder
             ↓ 通过
        Validator / Typecheck / Build
             ↓ 失败
        Repair（最多一次）
             ↓
        Commit / Promote
```

每个 Agent 有独立的 system prompt、模型配置和可用 Tool 集合。Worker 作为编排器决定状态流转，Agent 不直接调用校验/构建/Git 等关键动作。

#### 方向二：分角色支持与用户可指定角色

用户在工作台输入框中可以显式指定角色或自动触发角色：

- `@architect 重新设计状态管理，把 local state 抽到 hooks`
- `@builder 给 TodoList 增加按优先级排序`
- `@repair 这个类型错误怎么修`
- `@reviewer 看看这段代码有没有问题`

实现要点：

- 输入框解析 `@role` 前缀，映射到对应 Agent；
- 不同角色使用不同 system prompt 和 Tool 集；
- 角色切换不改变安全边界——所有写操作仍通过受限文件 Tool，校验/构建仍由 Worker 执行；
- 角色状态作为 Run 元数据持久化，便于后续审计和 Eval。

#### 方向三：Agent 记忆、自评

1. **项目级记忆**
   - 记录项目常用模式、用户偏好（如“总是用 zustand 管理状态”、“偏好函数组件”）；
   - 每次 Run 把记忆作为 soft prompt 注入，Agent 输出更一致。

2. **对话摘要**
   - 长对话超过模型上下文时，自动压缩历史为摘要；
   - 摘要只保留用户目标、关键决策和未解决问题，不保留完整 Tool 轨迹。

3. **Agent 自评与互评**
   - Builder 输出后先由 Reviewer 评估：是否符合 Contract、是否有明显缺陷、是否引入未授权依赖；
   - Reviewer 给出“通过/不通过 + 原因”，不通过时自动回到 Builder；
   - 最多一次 Reviewer 循环，避免无限循环。

### 4.5 为什么这样排序

- **Agent 上下文是最高 ROI 投资**：`@文件` 和行级选中不需要大改架构，但能直接降低 Token 消耗、减少幻觉、提高增量修改成功率。
- **多 Agent 分角色是质量放大器**：把“一个人做所有事”变成“多个专家协作”，每个 Agent 的 Prompt 和 Tool 更聚焦，复杂需求拆解更合理。
- **自评/互评降低 Repair 率**：在提交前发现问题，比构建失败后再 Repair 成本更低。
- **平台验收项紧随其后**：版本 Diff/恢复、访客模式、导出是 Demo 可交付性的底线，必须在 Agent 升级基本成型后补齐。
- **基础设施和长期边界靠后**：Storage 适配、清理、指标是平台稳定运行的基础，但短期内本地卷足以验证核心逻辑；自定义依赖、多页面、Race Mode 会显著扩大安全边界，ROI 不高，放在最后。

## 六、总结

Lite Atoms Dev 的核心价值在于：**用一个边界清晰、步骤真实、失败可恢复的工程闭环，证明 AI Coding 不仅可以“生成代码”，还可以“可信地交付代码”。**

当前 P0 主链路（计划、审批、生成、校验、构建、提交、预览、取消、一次修复）已经实现并稳定运行；Plan/Build 双模式、代码引用上下文（@文件与行级选中）、版本只读回溯、消息持久化等扩展能力也已落地，前后端检查全部通过。如果继续投入，**首要任务仍然是升级 Agent 的协作方式与上下文利用深度**：多 Agent 分角色（Architect / Reviewer）、Reviewer 自评互评、Builder 增量修改的结构化输出，这些优化对生成质量和用户体验的边际提升最大。在此基础之上，再补齐版本 Diff/恢复、访客模式、导出等验收项，并逐步增强可信闭环与可观测性，而不是过早扩展生成目标或开放自定义能力。
