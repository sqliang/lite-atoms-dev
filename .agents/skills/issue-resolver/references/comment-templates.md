# Issue 评论模板

Step 6 发布结论评论时，根据决策结果选择对应模板。

---

## 模板 A：自动实现报告

当 Step 4 评估通过、已自动完成分支 → 改代码 → 提交 → PR 后，用此模板发布评论：

```markdown
## AI 自主实现报告

### 自我评估

<!-- 根据 complexity-criteria.md 逐项说明为什么判断可以自动实现 -->
<!-- 例如：涉及文件仅 2 个（≤5）、优先级 P2（非 P0）、改动为简单配置值替换、无跨模块影响、无新增依赖 -->

### 改动总结

- **分支**: `fix/xxx` / `feat/xxx`
- **PR**: [#N](https://github.com/.../pull/N)
- **改动文件**: N 个
- **改动内容**: 一句话描述做了什么

### 验证结果

<!-- 列出 typecheck / build / lint 等自动化检查的结果 -->
- `pnpm typecheck` — 通过，零错误
- `pnpm build` — 通过

### PR 链接

<!-- 粘贴 PR URL，便于用户一键跳转 review -->
```

---

## 模板 B：解决方案（需人工处理）

当 Step 4 评估不通过、决定不自动实现时，用此模板发布评论：

```markdown
## 解决方案

<!-- 嵌入按 solution-template.md 格式输出的完整方案 -->
<!-- 包括：根因分析、涉及文件、具体改动、验证步骤、风险与回滚、影响范围 -->

### 自我评估结论

<!-- 说明为什么不能自动实现，具体哪些条件不满足 -->
<!-- 例如：涉及 6+ 个文件（超过 5）、需要修改 API 契约、P0 紧急问题、涉及 auth 逻辑 -->

<!-- 给出人工处理的建议和注意事项 -->
```

---

## 重要

- **两个模板二选一**，每次只发布一条评论
- 评论内容使用中文（与项目注释规范一致）
- 发布前确保所有链接（PR URL、Issue URL）有效
