# 复杂度评估标准

在制定方案后，按此清单逐项自评，决定 **自动实现** 还是 **仅评论**。

---

## 自动实现条件（必须全部满足）

- [ ] 涉及文件数 ≤ 5
- [ ] 无需数据库迁移或数据格式变更
- [ ] 不改变外部 API 契约（REST API、CLI 接口、配置文件 schema）
- [ ] 不涉及认证 / 授权 / 安全相关代码
- [ ] Issue 优先级为 P1 / P2 / P3（非 P0）
- [ ] 改动是增量的或简单替换（非大规模重构逻辑结构）
- [ ] 跨模块级联影响 ≤ 3 个下游（通过 grep 确认）
- [ ] 不需要新增第三方依赖

## 仅评论条件（满足任意一条即 defer）

- [ ] 涉及文件数 ≥ 6
- [ ] 需要数据库迁移
- [ ] 改变 API 契约或配置文件 schema
- [ ] 涉及认证 / 授权逻辑
- [ ] P0 紧急问题（需人工判断风险窗口）
- [ ] 大规模重构（改动逻辑结构，非简单替换）
- [ ] 跨模块级联影响 > 3 个下游
- [ ] Issue 正文明确说 "needs discussion" / "需要讨论" / "需要设计" 等

## 灰色地带处理

当条件处于模糊地带时（例如 4-5 个文件但其中一个改动较大，或 3 个下游但其中一个为核心模块），**倾向于 defer 给人工**。

**原则**：宁可错过自动化机会，也不自动执行有风险的改动。安全 > 效率。
