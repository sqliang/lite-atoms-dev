# 代码健康检查技能

对代码库进行全自动、多维度的健康检查，收集问题并输出结构化报告。

## 核心原则

**全自动执行，零交互。** 启动后不询问任何问题，不等待用户确认。遇到模糊情况按最合理的默认方式处理，继续向前推进，直至报告写入完成。

**只诊断，不修改。** 绝对不修改任何源代码文件。只读取、分析、记录问题。

## 执行流程

### Phase 1 — 确定扫描范围

从用户输入中提取范围信息：

- 未指定范围 → 扫描整个仓库
- 指定了目录（如 `pipeline/`）→ 只扫描该目录
- 指定了文件（如 `src/app/page.tsx`）→ 只扫描该文件

**始终跳过的目录**: `node_modules/`、`.git/`、`__pycache__/`、`.next/`、`data/`（含子目录）、`dist/`、`build/`、`.turbo/`

**始终跳过的文件**: 锁文件（`pnpm-lock.yaml`、`uv.lock`）、二进制文件、图片、字体、`*.pyc`、`*.ico`、`*.svg`、`*.woff*`

### Phase 2 — 广度发现

1. 列出扫描范围内的所有源文件（`find` 或 `git ls-files`）
2. 按语言分类：Python (`.py`)、TypeScript/TSX (`.ts`/`.tsx`)、YAML (`.yaml`/`.yml`)、JSON (`.json`)、Markdown (`.md`)、其他配置
3. 计算文件总数。超过 80 个文件时启动采样策略：
   - 核心目录（`pipeline/core/`、`pipeline/schemas/`、`src/lib/`、`src/app/`、`src/components/`）全部扫描
   - 叶子目录（`pipeline/ingestion/parsers/`、`pipeline/analysis/prompts/`）按代表性抽样
   - 每个目录至少扫描 3 个文件，不足则全扫

### Phase 3 — 逐维度深读分析

按以下六个维度依次扫描。每个维度独立进行，读完一批文件后立即在内存中记录发现（不要等到最后）。

#### 维度 1: 代码质量与潜在 Bug

逐文件检查的重点：

- **空值风险**: 可能为 None/undefined 的变量未经检查即使用；可选链缺失
- **异常处理**: try-except 吞掉异常（`except: pass`）；Promise 未 catch；异步函数未 await
- **类型安全**: Python 函数缺少类型标注（关键函数）；TypeScript 使用 `any` 而非具体类型
- **边界条件**: 数组/列表越界；除零未保护；字符串截取未检查长度
- **逻辑缺陷**: 条件互斥永不执行的分支；异步操作顺序错误；状态更新时机不对
- **React 特定**: useEffect 缺少依赖项或依赖项过多；key 使用 index；ref 未清理

#### 维度 2: 安全漏洞

- **注入风险**: SQL/命令拼接（非参数化）；`eval()`/`exec()`/`new Function()`；`dangerouslySetInnerHTML`
- **敏感信息**: API Key / Token / Password 硬编码在源码中；`.env` 文件未 gitignore
- **输入校验**: 用户输入未校验即使用；文件路径未防目录穿越
- **依赖安全**: `package.json` 或 `requirements.txt` 中依赖是否有已知 CVE（标注需人工确认）

#### 维度 3: 架构与设计

- **循环依赖**: Python 模块间循环 import；TypeScript 循环引用
- **职责混乱**: 单个文件超过 500 行（God File）；单个函数超过 80 行；组件既管数据又管渲染又管状态
- **耦合问题**: 模块直接依赖另一模块的内部实现而非公共接口；硬编码文件路径
- **抽象缺失**: 重复代码块（同一模式出现 3 次以上）；魔法数字未提取为常量

#### 维度 4: 性能问题

- **N+1 查询**: 循环内 await；循环内数据库/API 调用
- **React 渲染**: 未 memo 的大列表；未用 `useMemo`/`useCallback` 的计算密集型函数；Context 粒度过大
- **内存泄漏**: 未清理的定时器/订阅；闭包持有大对象；Python 循环引用未处理
- **同步阻塞**: Node.js 中的同步文件操作（`readFileSync`）；Python 中 CPU 密集任务阻塞事件循环

#### 维度 5: 工程规范

- **注释文档**: 导出函数/类缺少 docstring/JSDoc（对照项目 CLAUDE.md 规范）
- **命名规范**: 变量名与含义不符；缩写不清晰；常量未用大写；文件命名不一致
- **类型标注**: Pydantic Field 缺少 `description=`；TypeScript interface 缺少注释
- **导入组织**: 未分组 import；未清理的未使用 import

#### 维度 6: 技术债

- **标记注释**: `TODO`、`FIXME`、`HACK`、`XXX`、`WORKAROUND` — 记录位置和内容
- **硬编码**: 直接写在代码中的配置值（URL、端口号、超时时间）而非环境变量或配置文件
- **废弃使用**: 调用已标记为 deprecated 的 API；使用过时的 React 模式（如 class component 混用）
- **依赖过时**: 检查 `package.json` 和 `requirements.txt`，标注主版本号落后 2+ 个大版本的依赖

### Phase 4 — 汇总与去重

1. 将六个维度的发现合并
2. 去重规则：
   - 同一文件、同一行、同一维度的问题合并为一条（保留更具体的描述）
   - 同一文件、不同行但本质相同的模式问题合并为一条，列出所有位置
3. 为每条问题分配严重度：

| 严重度 | 判定标准 |
|--------|---------|
| **Critical** | 安全漏洞；确定会导致运行时 crash 的 bug；数据丢失风险；`eval()`/`exec()` 使用 |
| **High** | 潜在 bug（高触发概率）；重要异常处理缺失；N+1 查询；内存泄漏 |
| **Medium** | 代码规范违反；缺少类型标注；中等复杂度的重构建议；TODO/FIXME |
| **Low** | 命名建议；注释缺失；import 整理；轻微代码异味 |

4. 按严重度排序（同严重度内按文件路径排序）

### Phase 5 — 写入报告

将报告写入 `docs/issues/code-health-{YYYY-MM-DD-HHmmss}.md`。

**必须先确保 `docs/issues/` 目录存在**（不存在则创建）。

## 注意事项

- 每个维度读文件时，不要读完一个维度全部文件再进行下一个维度。一次打开的文件在分析时尽量跨维度复用，减少重复读取。
- 代码片段要包含足够的上下文（前后各 2-3 行），但不能太长（最多 15 行）。
- 如果某个维度没有发现问题，在报告中注明"未发现明显问题"，不要留空。
- 如果扫描范围超过 200 个文件，报告开头加一句提示："文件量较大，采用分层抽样策略，已扫描 {N} 个代表性文件。"
- 全部完成后，在对话中简要告知用户报告路径和问题总数摘要。
